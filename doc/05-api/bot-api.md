# Bot API

> Публичный API для сторонних разработчиков. Стиль — метод-ориентированный RPC (как Telegram Bot API), схема `schema/openapi/bot-api.yaml` — источник истины ([ADR-0009](../adr/0009-schema-first-api.md)). Модель данных — [data-model.md](../02-architecture/data-model.md) §12.

## 1. Принципы

1. **Бот — не пользователь.** У него нет @username, профиля и личных чатов. Бот — приложение с токеном, **установленное** в группу/канал/сообщество; аудитория видит сущность, не бота.
2. **Боты пишут только от имени сущностей.** Сущности не пишут в личку ([concept.md](../01-product/concept.md)) → бот физически не может написать личное сообщение. Сообщения сущности пользователю — карточки в окне 4, под контролем пер-сущностных настроек пользователя (уведомления / сообщения — и/или).
3. **Только публичный контур.** Боты никогда не получают ключей (GK, identity): установка в личные E2E-группы запрещена. Компрометация токена не раскрывает ни одного защищённого сообщения ([content-security-matrix.md](../01-product/content-security-matrix.md)).
4. **Два вида обращений.** Обращения к **виртуальному пользователю** — E2E (identity-ключ ВП), ботам **недоступны** (только люди-операторы). Обращения к **сущности** (группа/канал/сообщество) — публичный контур (plaintext-тред с честной пометкой в UI отправителя); их и обрабатывает scope `inbox`.
4. Совместимость: только аддитивные изменения, deprecation ≥ 6 месяцев.

## 2. Жизненный цикл

```
1. Создание: настройки приложения → «Боты» → [Создать бота] (имя, описание)
   → токен показывается ОДИН раз (hash на сервере)
2. Установка: владелец/админ сущности устанавливает бота с выбором scopes
   (экран согласия: «бот сможет: отправлять сообщения, модерировать…»)
3. Работа: getUpdates (long polling) или webhook
4. Отзыв: установка отзывается сущностью мгновенно; токен — владельцем бота
   (ротация: новый токен, старый гаснет)
```

## 3. Scopes (права установки)

| Scope | Даёт | Где применимо |
|-------|------|---------------|
| `send_messages` | Сообщения от имени группы | Публичная группа |
| `moderate` | Удаление сообщений/комментариев, бан/разбан участников | Группа, канал (комментарии), сообщество |
| `publish_posts` | Создание/правка/удаление **своих** постов (при премодерации канала → pending) | Канал |
| `invite_links` | Создание/отзыв инвайт-ссылок (срок/лимит/роль) | Группа, канал, сообщество |
| `inbox` | Чтение и ответы на **публичные** обращения к сущности (E2E-обращения к ВП ботам недоступны) | Все |
| `notify` | `notifyUser` — карточка пользователю в окно 4 | Все, с жёсткими лимитами (§6) |

## 4. Аутентификация

- Токен в пути: `POST /bot/v1/{token}/{method}`; TLS обязателен.
- **HMAC-подпись** тела запроса заголовком `X-Bot-Signature` (ключ выдаётся вместе с токеном) — защита от утечки токена в логах прокси.
- Webhook: сервер подписывает доставку заголовком `X-Webhook-Secret` (задаётся при `setWebhook`); бот обязан проверять.
- Хранение токенов на стороне разработчика — секрет-менеджер; ротация поддерживается API (`rotateToken`).

## 5. Updates

Конверт: `Update {update_id: int64, type, payload}` — монотонный `update_id`, доставка **at-least-once** (боты обязаны быть идемпотентными по `update_id`).

| Тип | Когда | Требуемый scope |
|-----|-------|-----------------|
| `group_message` | Новое сообщение в публичной группе | send_messages или moderate |
| `comment.new` | Комментарий к посту канала | publish_posts или moderate |
| `post.reaction` | Эмоция/[+]/[−] на пост | publish_posts |
| `member.joined` / `member.left` | Изменение участников (группа, сообщество; у канала аудитория — подписки, приходит агрегат `subscriber.count`) | moderate |
| `moderation.request` | Пост ждёт премодерации; жалоба | moderate |
| `inbox.appeal` | Пользователь написал сущности (обращение) | inbox |

Получение:
- `getUpdates {offset, limit, timeout}` — long polling (для разработки, без публичного IP);
- `setWebhook {url, secret}` — push с retry (экспоненциальный backoff, TTL 24ч);
- механизмы взаимоисключающие (как у Telegram).

## 6. Методы (ядро)

| Метод | Действие | Scope |
|-------|----------|-------|
| `getMe` / `getInstallations` | Инфо о боте и установках | — |
| `sendGroupMessage {group_id, text, entities?, reply_to?}` | Сообщение в группу | send_messages |
| `editMessage` / `deleteMessage` | Правка/удаление (своих — всегда; чужих — moderate) | send_messages / moderate |
| `publishPost {channel_id, kind, title?, body, entities, media_ids, attribute_ids}` | Пост канала (формат — [module-boundaries.md](../02-architecture/module-boundaries.md) §5) | publish_posts |
| `editPost` / `deletePost` | Правка/удаление своих постов | publish_posts |
| `approvePost` / `rejectPost` | Решения премодерации (согласовано с update `moderation.request`) | moderate |
| `banMember {target, user_id, until?}` / `unbanMember` | Модерация участников; для канала = запрет комментирования (запись в memberships только ради `banned_until`) | moderate |
| `createInviteLink {target, expires?, max_uses?, role?}` / `revokeInviteLink` | Инвайт-ссылки | invite_links |
| `answerAppeal {thread_id, text}` / `closeAppeal` | Ответ на **публичное** обращение к сущности; у пользователя атрибутируется сущности (byline «Канал X», факт бота — в аудите) | inbox |
| `notifyUser {user_id, text, link?}` | Карточка в окно 4 пользователя | notify |
| `uploadMedia` → presigned | Медиа для постов/сообщений (только plaintext, публичный контур) | по контексту |

## 7. Лимиты и анти-спам

| Ограничение | Значение (первый релиз Bot API) |
|-------------|----------------|
| Запросы на токен | 30 rps burst / 10 rps sustained |
| `sendGroupMessage` в одну группу | 20/мин (наследует slow mode группы) |
| `notifyUser` | Только подписчикам/участникам сущности; уважает пер-сущностную настройку «сообщения: выкл»; ≤ 1 карточка пользователю от сущности в час; push-группировка карточек — **per сущность** (аналог правила «≤1 за 5 мин из одного чата») |
| Установок бота | 100 сущностей на бота |

Нарушения → 429 с `retry_after`; систематические → авто-заморозка токена + уведомление владельцу. Жалобы пользователей на сущность распространяются на действия её ботов.

## 8. Ошибки

`{ok: false, error_code, description, retry_after?}` — коды стабильны (часть контракта): `401 invalid_token`, `403 missing_scope`, `403 e2e_target` (попытка достучаться до защищённого контура), `429 flood`, `400 bad_request`.

## 9. SDK

Из схемы генерируются официальные SDK (Python, TypeScript). Высокоуровневый фреймворк в духе aiogram 3.x (Router, фильтры, middleware, FSM) — отдельный проект поверх HTTP API, пост-MVP ([ADR-0009](../adr/0009-schema-first-api.md)).
