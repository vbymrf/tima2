# Путь масштабирования

> Эволюция от одного VPS до распределённой системы. Каждый шаг включается **по триггеру**, а не заранее ([ADR-0003](../adr/0003-postgresql-storage.md)). Telegram-паттерны из исследований (шардинг по chat_id, hot/cold, append-only) заложены в модель данных с первого дня — здесь описано, когда их «включать».

## Ступень 0 — один VPS (MVP, бета)

Всё в одном docker-compose ([server-setup.md](./server-setup.md)). Держит порядка 10k активных пользователей / 10k одновременных WS.

## Ступень 1 — разделение узлов

**Триггер:** CPU > 70 % устойчиво, либо диск БД конкурирует с медиа.

```
VPS-1: Caddy + backend + worker         VPS-2: PostgreSQL + Redis
VPS-3: MinIO                            VPS-4: LiveKit + TURN
```

- Caddy upstreams → добавление второго backend-узла без простоя (backend stateless, WS-состояние в Redis pub/sub).
- PostgreSQL: PgBouncer перед базой; streaming-реплика для чтения и failover.

## Ступень 2 — горизонтальный backend и очереди

**Триггер:** > 50k одновременных WS, отставание fan-out.

- N backend-узлов за Caddy (sticky по device для WS не нужен — события через Redis pub/sub; при росте — NATS/Redis Cluster).
- Fan-out лент: Redis Streams → **Kafka** (durable, партиции по user_id), воркеры масштабируются группами.
- Push-очередь — отдельные воркеры.

## Ступень 3 — шардинг данных

**Триггер:** таблицы сообщений > ~1 ТБ или деградация p95 записи.

- Включается реальный `GetShard(chatID)`: несколько PostgreSQL-кластеров, чат целиком живёт на своём шарде (нет распределённых транзакций).
- `feeds` шардируются по user_id (независимо от сообщений — это разные нагрузки, как в исследованиях: «личные сообщения и лента — два разных приложения»).
- Hot/warm/cold для сообщений: старые партиции → архивные таблицы/дешёвые диски; медиа уже управляется S3 Lifecycle.

## Ступень 4 — специализированные подсистемы

| Подсистема | Когда | Что |
|-----------|-------|-----|
| Поиск | p95 > 500 мс / >10M документов | PG FTS → **OpenSearch** ([ADR-0007](../adr/0007-search-split.md)); фасад API не меняется |
| Gateway | 3+ выделенных сервиса | Caddy → **Envoy** (mTLS между сервисами, gRPC) ([ADR-0008](../adr/0008-caddy-edge.md)) |
| Оркестрация | > ~5 узлов, частые деплои | docker-compose → **Kubernetes** (может не понадобиться долго) |
| SFU | Латентность межрегиональных звонков | Региональные LiveKit-узлы, выбор по гео |
| MinIO | Отказоустойчивость хранилища | Distributed MinIO (erasure coding, 4+ узла) |
| Гео-распределение | Юридические/латентные требования | Кластеры по регионам, асинхронная репликация |

## Что уже готово к масштабу (не переделывается)

- Append-only сообщения + месячные партиции → перенос/архивирование партиций без миграции модели.
- Интерфейс `GetShard(chatID)` во всех запросах к сообщениям.
- Presigned-доступ к медиа → трафик файлов не проходит через backend изначально.
- Stateless backend + события через Redis → добавление узлов конфигурацией.
- Клиент — основной хранитель приватной истории → сервер не обязан хранить всё вечно (retention, [escrow-legal-access.md](../03-security/escrow-legal-access.md) §5).

## Чего сознательно НЕ делаем заранее

Kafka, OpenSearch, Kubernetes, Envoy, шардинг, RocksDB/самописные хранилища, микросервисная нарезка — всё по триггерам выше. Ранняя сложность — главный риск маленькой команды.
