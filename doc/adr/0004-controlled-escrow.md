# ADR-0004: Клиентское шифрование с controlled escrow

**Статус:** принят · **Дата:** 2026-07-12

## Контекст

Чистый Signal Protocol (X3DH + Double Ratchet как единственный механизм) создаёт операционные проблемы: потеря ratchet-состояния = потеря переписки, хрупкость при офлайне и мультиустройстве, невозможность юридического доступа. Подробный разбор — [messenger-crypto-architecture.md](../messenger-crypto-architecture.md) §2–4.

## Решение

Гибрид **«Конверт + Wrapped Keys + Escrow»** (Kodium):

1. **Конверт** — контент всегда шифруется на клиенте случайным `message_key` (SecretBox).
2. **Wrapped keys (план Б)** — `message_key`, завёрнутый в identity-ключ каждого получателя/устройства, хранится на сервере → надёжная асинхронная доставка и мультиустройство.
3. **Escrow** — `message_key`/`GK` дополнительно инкапсулируется ML-KEM-768 на публичный ключ escrow; приватный ключ — в HSM/анклаве, доступ только M-of-N по юридическому запросу с audit log.
4. **Double Ratchet** — опциональный слой PFS для активных сессий (путь A); при его отказе работает путь B.

Шифруется **всё, кроме публичного контента**: личные чаты, личные группы (Sender Keys/GK), личные коллекции, голосовые сообщения, вложения приватных чатов. Публичные группы, каналы, публичные медиа — plaintext ([content-security-matrix.md](../01-product/content-security-matrix.md)).

## Отклонено

- Чистый Signal — хрупкость доставки, нет escrow.
- Только статический Box без ratchet — нет forward secrecy.
- Отложить шифрование на «после MVP» — миграция форматов сообщений и БД дороже, чем слои с первого дня.
