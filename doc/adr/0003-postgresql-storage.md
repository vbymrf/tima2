# ADR-0003: PostgreSQL 16 как основное хранилище

**Статус:** принят · **Дата:** 2026-07-12

## Контекст

В исследованиях рассматривались RocksDB и самописное Telegram-подобное хранилище «на миллиарды сообщений». Это оптимизации под масштаб, которого на старте нет.

## Решение

- **PostgreSQL 16** — единственная БД: пользователи, чаты, сообщения (ciphertext), ключи (wrapped/escrow blobs), группы, ленты, коллекции, реакции.
- **Redis** — кэш, online-статусы, Pub/Sub для WS-нотификаций, очереди (Redis Streams).
- **MinIO** (S3 API) — все бинарные данные; в PostgreSQL только метаданные ([media-storage.md](../04-data/media-storage.md)).
- Telegram-паттерны закладываются в **схему и интерфейсы**, не в инфраструктуру: append-only сообщения, партиционирование по времени, интерфейс шардинга `GetShard(chatID)` (на MVP всегда возвращает единственный узел).

## Отклонено (отложено)

- RocksDB / самописное KV-хранилище — преждевременная оптимизация.
- Cassandra, Kafka на старте — см. [scaling.md](../07-deployment/scaling.md), подключаются по триггерам нагрузки.

## Последствия

- Схема БД — [data-model.md](../02-architecture/data-model.md).
- Сообщения не редактируются на уровне строк (append-only + флаги soft delete) — это сохраняет путь миграции на специализированное хранилище без смены модели.
