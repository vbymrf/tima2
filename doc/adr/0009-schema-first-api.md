# ADR-0009: Schema-first API и два контура (Client / Bot)

**Статус:** принят · **Дата:** 2026-07-12

## Контекст

API мессенджера потребляют три стороны: наши KMP-клиенты, воркеры бэкенда и сторонние разработчики ботов. Ручная синхронизация типов между Go, Kotlin и SDK ботов гарантированно расходится. Урок aiogram 3.x: его типы генерируются из схемы Telegram Bot API, а не пишутся руками.

## Решение

1. **Схема — источник истины.** Каталог `schema/` (в монорепо): `openapi/client-api.yaml`, `openapi/bot-api.yaml`, `proto/*.proto`, `proto/ws-events.proto`. Всё остальное генерируется: Go-хендлеры и типы (oapi-codegen/protoc), KMP-клиент, Bot SDK (Python/TS), портал документации.
   - Уже зафиксировано: `proto/envelope.proto` + `proto/message_body.proto` (конверт сообщения) с определением `canonical_bytes` ([proto/README.md](../../schema/proto/README.md)) и каноничные тест-векторы ([test-vectors/](../../schema/test-vectors/README.md)) — контракт крипто-ядра, воспроизводимый независимой реализацией (tweetnacl-js).
2. **Правка API = правка схемы** (PR со схемой); CI проверяет обратную совместимость (oasdiff) и регенерирует артефакты.
3. **Два контура с разными гарантиями:**

| | Client API | Bot API |
|---|---|---|
| Стиль | REST `/api/v1` | RPC `/bot/v1/{token}/method` ([bot-api.md](../05-api/bot-api.md)) |
| Стабильность | Меняем свободно (владеем обоими концами) | Контракт: только аддитивные изменения; deprecation-цикл ≥ 6 месяцев |
| Auth | Device JWT + аттестация | Bot token + HMAC + webhook secret |
| Область | Всё | Только публичный контур, права по scopes |

## Отклонено

- Ручные типы на каждой стороне — источник рассинхрона.
- Единый контур для клиентов и ботов — публичный контракт заморозил бы внутренние итерации.
- REST-стиль для Bot API — метод-ориентированный RPC знаком экосистеме Telegram-ботов (портирование привычек и фреймворков дешевле).
