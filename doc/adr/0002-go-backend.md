# ADR-0002: Бэкенд на Go

**Статус:** принят · **Дата:** 2026-07-12

## Контекст

Бэкенд — сетевые сервисы: REST API, WebSocket-шлюз, воркеры очередей, fan-out ленты. Рассматривались Go и Kotlin/Ktor (единый язык с клиентом, доступ к Kodium на сервере).

## Решение

Весь серверный код — **Go**.

- Лучший баланс производительности и скорости разработки для сетевых сервисов (goroutines для WS-соединений, низкое потребление памяти).
- Богатая экосистема инфраструктуры (pgx, go-redis, minio-go, LiveKit server SDK на Go).
- Серверу **не нужен Kodium**: по архитектуре ([crypto-protocol.md](../03-security/crypto-protocol.md)) сервер хранит только ciphertext и wrapped keys; вся криптография — на клиенте. Проверка подписей Ed25519 доступна в stdlib Go.

## Отклонено

- **Kotlin/Ktor** — общие DTO с клиентом привлекательны, но контракт фиксируется схемами (Protobuf/OpenAPI), а не общим кодом; JVM тяжелее в эксплуатации на маленьких VPS.
- Гибрид Go+Kotlin — два рантайма на старте не оправданы.

## Последствия

- Контракты клиент↔сервер описываются схемами: Protobuf для сообщений, OpenAPI для REST ([05-api](../05-api/api-overview.md)).
- Начинаем с **модульного монолита** (один бинарник, чёткие внутренние границы сервисов), выделение микросервисов — по мере нагрузки ([scaling.md](../07-deployment/scaling.md)).
