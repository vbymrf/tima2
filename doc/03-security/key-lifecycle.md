# Жизненный цикл ключей

> Генерация, хранение, привязка устройств, восстановление, отзыв. Ключи и их роли — [crypto-protocol.md](./crypto-protocol.md) §2. UI: [22-auth-registration](../doc_UI/22-auth-registration.md), [23-auth-recovery](../doc_UI/23-auth-recovery.md), [24-device-linking](../doc_UI/24-device-linking.md).

## 1. Регистрация (первое устройство, полный аккаунт)

```
1. Телефон подтверждён SMS + привязан email (обязателен)
2. Устройство генерирует одну `Kodium.generateKeyPair()` → `KodiumPrivateKey`
   (даёт и X25519-encryption, и Ed25519-signing из общего seed)
   — приватный ключ в Keystore (Android) / Secure Enclave (iOS)
3. Аттестация: Play Integrity / App Attest ([client-attestation.md](./client-attestation.md))
4. Регистрация device_id + публичных ключей на сервере (таблица devices)
5. Первое телефонное устройство помечается is_trust_anchor = TRUE
6. Генерация 10 резервных кодов (для восстановления входа) — показываются один раз
7. Ключ локальной БД: производный от пароля пользователя (PBKDF2, Kodium) — вводится раз в сессию
```

## 2. Привязка нового устройства

Телефон — **якорь доверия**. Схема для Windows/Desktop (и вторых телефонов):

```
Новое устройство                     Сервер                Доверенный телефон
  1. генерирует identity/signing        │                         │
  2. показывает QR {device_pub, nonce}  │                         │
  │                                     │   3. скан QR камерой ───│
  │                                     │◄── 4. POST /link/confirm │
  │                                     │      (подпись телефона:  │
  │                                     │       «доверяю ключу X») │
  │◄── 5. device зарегистрирован ───────│                         │
  6. доверенный телефон ПЕРЕЗАВОРАЧИВАЕТ (re-wrap) ключи сообщений
     последних 90 дней на identity нового устройства (путь B);
     сервер сам не может — wrapped keys per-device
  7. опционально: телефон передаёт локальную историю старше 90 дней
     по зашифрованному каналу (Box между устройствами)
```

Windows-клиент не проходит аппаратную аттестацию — его доверие производно от телефона ([client-attestation.md](./client-attestation.md)).

## 3. Мультиустройство в повседневной работе

- Каждое сообщение получает wrapped_key для **всех активных устройств** отправителя и получателя ([crypto-protocol.md](./crypto-protocol.md) §3.2).
- Ratchet-сессии по-устройствные; их синхронизация между устройствами не выполняется (устройство без сессии использует путь B).
- Список устройств виден пользователю; «выйти со всех устройств» — массовый отзыв (§5).

## 4. Восстановление

| Сценарий | Что происходит |
|----------|---------------|
| Забыт пароль входа | Восстановление через SMS + email + резервный код; ключи устройств не затрагиваются |
| Потерян телефон, есть другое доверенное устройство | Второе устройство становится якорем; старый device отзывается |
| Потеряны все устройства | Новый вход по SMS+email+резервному коду → аккаунт восстановлен, но **защищённая история недоступна**: wrapped keys адресованы identity-ключам утраченных устройств, перезавернуть их некому, escrow пользователю недоступен (осознанное свойство схемы) |
| Забыт пароль локальной БД | Локальный кэш сбрасывается; история восстанавливается как «потеряны все устройства» |

Восстановления «через escrow» для пользователей **нет** — escrow только для юридической процедуры ([escrow-legal-access.md](./escrow-legal-access.md)).

## 5. Отзыв и компрометация

```
Отзыв устройства (revoked_at):
1. Сессии и push-токен аннулируются; WS разрывается
2. Wrapped keys для устройства перестают создаваться; существующие помечаются deleted
3. Во всех личных группах пользователя — ротация GK (устройство больше не получит новых версий)
4. Собеседникам 1:1 — событие key.changed (UI-предупреждение при смене identity)
```

- Компрометация identity-ключа = отзыв устройства + перегенерация на чистом устройстве.
- Смена identity собеседника всегда видима в UI (safety number) — защита от подмены сервером.

## 6. PreKeys (фаза ratchet)

- Каждое устройство поддерживает на сервере: 1 Signed PreKey (ротация раз в 7 дней) + пул 100 One-Time PreKeys (пополнение при остатке < 20).
- Signed PreKey обязан быть верифицирован подписью identity при handshake (условие gate, [ADR-0005](../adr/0005-kodium-readiness-gate.md)).
- Consumed OTK помечаются и не переиспользуются; при исчерпании X3DH деградирует на Signed PreKey (штатно).

## 7. Временный режим (аккаунт без телефона/email)

Пользователь может начать работу **без указания телефона и email** ([22-auth-registration](../doc_UI/22-auth-registration.md)).

| Свойство | Временный аккаунт |
|----------|-------------------|
| Идентификация | Только device_id + локальный пароль; `account_type = 'temporary'` |
| Ключи | Identity/signing генерируются как обычно; крипто-протокол идентичен полному аккаунту (конверт + wrapped keys + escrow) |
| Устройства | Ровно одно; QR-привязка недоступна (нет якоря доверия) |
| Восстановление | **Отсутствует.** Потеря устройства/пароля = потеря аккаунта |
| Поиск по номеру | Невозможен (номера нет); добавление в контакты — по QR/username |
| Срок жизни | Удаление после 30 дней неактивности (`last_active_at`) с предупреждением |
| Апгрейд | `POST /auth/upgrade`: привязка телефона + email → выдача резервных кодов → аккаунт становится полным, устройство — якорем доверия. Ключи и история не меняются |

Ограничения против злоупотреблений (антиспам): пониженные rate-лимиты, аттестация обязательна, невозможность создавать публичные группы/каналы до апгрейда.

## 8. Ключи виртуальных пользователей

У ВП нет собственных устройств ([virtual-users.md](../01-product/virtual-users.md)) — его ключами владеют устройства команды:

```
1. Создание ВП: устройство владельца генерирует identity + signing пары ВП
2. Приватные ключи существуют ТОЛЬКО wrapped на устройства владельца и операторов
   (vu_wrapped_keys, версия ключа); в Keystore не кладутся — расшифровка по требованию
3. Входящее к ВП: обычный конверт, wrapped_key на identity-ключ ВП
4. Исходящее от ВП: подпись signing-ключом ВП; actor_user_id — в серверном аудите
5. Смена состава операторов → ротация (новая версия ключей, перевыдача wrapped)
6. Передача ВП: ротация обязательна по умолчанию — старый владелец гарантированно
   не читает сообщения ПОСЛЕ передачи (копии, сделанные до неё, неустранимы)
```

Ограничение MVP: ВП не участвует в звонках и не имеет ratchet-сессий (только конвертный путь B).

## 9. Ротации (сводная таблица)

| Ключ | Ротация |
|------|---------|
| message_key | Каждое сообщение (одноразовый) |
| GK | 100 сообщений / join / leave / компрометация |
| media_key | Каждый объект |
| Signed PreKey | 7 дней |
| Identity/signing устройства | Только при компрометации (= новое устройство) |
| Ключи ВП | Смена операторов / передача владения (§8) |
| Escrow public | 12 месяцев ([escrow-legal-access.md](./escrow-legal-access.md) §6) |
| Секреты сервера (JWT, HMAC) | 90 дней; хранение в secret-менеджере ([server-setup.md](../07-deployment/server-setup.md)) |
