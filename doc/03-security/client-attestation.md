# Аттестация клиентов

> Задача: сервер принимает чувствительные запросы только от неизменённого официального приложения на реальном устройстве. Без аттестации остальная защита обесценивается (левый клиент может честно говорить по протоколу).

## 1. Матрица

| Платформа | Механизм | Что доказывает |
|-----------|----------|----------------|
| iOS | **App Attest** (DeviceCheck) | Запрос от неизменённого бинарника на реальном Apple-устройстве |
| Android | **Play Integrity API** | Легальное Android-устройство; приложение из Google Play / подписано нами |
| Windows | **QR-привязка через телефон** | Доверие производно от аттестованного мобильного устройства ([key-lifecycle.md](./key-lifecycle.md) §2) |
| Web (в планах) | **QR-привязка через телефон** | Та же модель, что Windows; плюс ограничения браузерного хранилища ключей |

Windows и Web аппаратно не аттестуются — не тратим усилия, телефон служит якорем доверия.

## 2. Интеграция в KMP

```kotlin
// commonMain
interface AttestationProvider {
    suspend fun getAttestationToken(action: String): String
}
// iosMain     → DCAppAttestService: генерация ключа при первом входе, подпись хеша запроса
// androidMain → Play Integrity: prepare при старте, request при важных вызовах
// desktopMain → LinkedDeviceProvider: подписанный телефоном линк-сертификат
```

**Бэкенд:** эндпоинты `POST /attest/ios`, `POST /attest/android` — верификация через Apple/Google API, результат кэшируется на устройство (TTL 24 ч, Redis). Аттестация требуется при: регистрации, привязке устройства, смене ключей, массовых операциях. Не на каждый запрос (латентность/квоты).

## 3. Защита API помимо аттестации

| Что | Как |
|-----|-----|
| Отличить клиент от скрипта | HMAC-подпись запросов (ключ обфусцирован в клиенте; ротация каждые 2 недели через force update) |
| Украденные ключи/токены | Rate limiting per device/user (Redis), мониторинг аномалий |
| Секреты третьих сторон | Не хранятся в клиенте вообще — только backend-proxy |
| Ключ локальной БД | Производный от пароля пользователя + привязка к устройству |

**Золотое правило:** считаем, что любые ключи из клиента будут извлечены. Значит: чувствительное — на сервере, лимиты — на сервере, ротация — регулярная, мониторинг — обязателен.

## 4. Уровни доступа по статусу аттестации

| Статус | Доступ |
|--------|--------|
| Аттестован | Полный |
| Аттестация недоступна (старое устройство, отсутствие GMS) | Ограниченный: вход разрешён, порог rate limit ниже, чувствительные операции — подтверждение с доверенного устройства |
| Провал аттестации (root/jailbreak-подделка, форк) | Отказ в регистрации/привязке; существующая сессия — только чтение + предупреждение |

Политика уровней настраивается серверно (kill switch на случай сбоев Google/Apple API).
