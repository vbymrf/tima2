# Escrow и юридический доступ

> Политика controlled escrow: зачем, как, с какими гарантиями и как это коммуницируется пользователям. Решение — [ADR-0004](../adr/0004-controlled-escrow.md), техника — [crypto-protocol.md](./crypto-protocol.md) §6.

## 1. Суть

Каждый ключ защищённого контента дополнительно инкапсулируется (ML-KEM-768) на публичный escrow-ключ. Приватный escrow-ключ существует только внутри HSM/анклава и может быть применён исключительно по формальной процедуре. Это позволяет исполнять юридически обязывающие запросы **без** компрометации identity-ключей пользователей и без бэкдора в протоколе.

## 2. Терминология для пользователей

Юридически схема **не является strict E2E**: существует технический канал доступа по процедуре.

| Где | Формулировка |
|-----|--------------|
| UI (статус чата) | «Защищённый чат — сообщения шифруются на вашем устройстве» |
| UI (окно информации, [16-profile-popup](../doc_UI/16-profile-popup.md)) | Краткое пояснение + ссылка на политику |
| Политика конфиденциальности | Полное описание: шифрование на клиенте; доступ провайдера возможен только по юридически обязывающему запросу через многостороннюю процедуру с аудитом |
| Маркетинг | Не использовать «E2E»/«сквозное» без оговорки |

Внутри документации термин «E2E» допустим только как UI-ярлык со ссылкой сюда.

## 3. Процедура доступа

```
1. Юридически обязывающий запрос (суд/уполномоченный орган) → Security Officer
2. Проверка объёма запроса: конкретный пользователь/чат + период (не bulk)
3. M-of-N (Shamir, минимум 3-of-5 держателей долей) → разблокировка операции в HSM
4. HSM: для каждого escrow_blob запрошенного периода → decapsulate → message_key / GK
5. Расшифровка payload'ов — в изолированной среде следственного ПО, не в продакшн-контуре
6. Append-only audit log: кто, основание, объём («N сообщений, из них M soft-deleted»)
```

**Инварианты:**
- Никакого автоматического или программного доступа из бэкенда к приватному ключу escrow.
- Держатели долей — разные роли/юрисдикции; ни один сотрудник не имеет доступа единолично.
- Audit log недоступен для редактирования (WORM-хранилище) и периодически сверяется.
- Ключи выдаются **по периоду**, не «мастер-ключ навсегда».

## 4. Soft delete

- «Удалить для всех» → `deleted = TRUE`: клиентам не отдаётся, физически сохраняется.
- Юридический доступ видит удалённые сообщения периода; факт удаления отражается в аудите.
- Физическое удаление — по retention-политике (§5) фоновыми джобами.

## 5. Retention и периоды ключей

| Период | Доступ клиента | Юридический доступ |
|--------|----------------|--------------------|
| 0–90 дней | wrapped keys на сервере (путь B) | escrow_blob |
| 90 дней – 7 лет | только локальная копия устройства | escrow_blob (архив) |
| > 7 лет | — | удалено (включая escrow_blob) |

- Wrapped keys старше 90 дней удаляются с сервера: история «живет» на устройствах пользователя ([sync-offline.md](../04-data/sync-offline.md)); новое устройство получает только последние 90 дней + опциональный перенос с доверенного устройства.
- Для групп и медиа escrow идёт по `gk_version` / `period_id` — один blob на период, не на файл.

## 6. Ротация escrow-ключа

- Плановая ротация Escrow Public: раз в 12 месяцев (новый ключ для новых сообщений; старые blobs остаются под старым ключом в HSM).
- Компрометация HSM-инфраструктуры → немедленная ротация + расследование + уведомление по применимому праву.

## 7. Стадии внедрения

| Фаза | Реализация |
|------|-----------|
| MVP / бета | Stub-анклав: изолированный контейнер, тот же API (encapsulate на клиенте — уже настоящий ML-KEM), доли Shamir у команды. Данные беты помечаются как подлежащие escrow-политике |
| Публичный релиз (gate) | Production HSM или AWS Nitro Enclave; формализованные держатели долей; юридическая проверка политики; WORM audit log |

## 8. Риски

| Риск | Митигация |
|------|-----------|
| Восприятие «escrow = backdoor» | Прозрачная публичная политика, M-of-N, аудит, честная терминология (§2) |
| Принуждение к bulk-доступу | Технически ключи выдаются по периодам/чатам; политика отклонения bulk-запросов |
| Компрометация держателей долей | Разные юрисдикции/роли, ротация долей |
| Утечка audit log | Лог не содержит контента, только метаданные доступа |
