# Виртуальные пользователи

> **Виртуальный пользователь (ВП)** — полноценный аккаунт, принадлежащий другому пользователю. Для всей системы **неотличим от обычного**: свой @username, профиль, личные чаты, рейтинг; может владеть каналами, группами, сообществами и публиковать контент. Отличия три: его сообщениями управляют с основного логина (окно 4), у него нет собственных устройств/пароля, и его можно **передать** другому владельцу.
> UI: [10-social-interaction](../doc_UI/10-social-interaction.md) (окно 4), блогерские окна [29](../doc_UI/29-blogger-media-window.md)/[30](../doc_UI/30-blogger-news-window.md). Схема — [data-model.md](../02-architecture/data-model.md) §10, ключи — [key-lifecycle.md](../03-security/key-lifecycle.md) §8.

## 1. Модель

```
ВП = строка в users:
    account_type = 'virtual'
    owner_user_id → основной аккаунт
    нет телефона / email / пароля / устройств / сессий
    всё остальное — как у любого пользователя
```

- **Единое пространство имён**: @username ВП живёт среди всех остальных.
- ВП может: вести личные чаты, создавать и владеть каналами/группами/сообществами, публиковать посты, назначаться на роли (`memberships` без изменений — ВП обычный user), получать эмоции и рейтинг «+/−» (**отдельный** от владельца — иначе счётчики раскроют владельца анонимного ВП).
- ВП не может (MVP): звонить и участвовать в звонках (голос раскрывает оператора и усложняет ключи), создавать собственных ВП, проходить аттестацию (наследует доверие устройств владельца/операторов).
- Блогерские «анонимные псевдонимы» — это ВП (прежняя сущность `aliases` упразднена).
- Связь «ВП → владелец» известна только серверу (и операторам); публично не раскрывается.

## 2. Управление: владелец и операторы

- **Владелец** — полный контроль: действия от имени ВП, назначение операторов, передача, удаление.
- **Операторы** — реальные пользователи, назначенные владельцем (например, редакторы канала, которым владеет ВП). Действуют от имени ВП в рамках роли. Суб-роль `manager` — оператор, которому владелец делегировал управление составом операторов.
- Каждое действие от имени ВП хранит серверно **`actor_user_id`** (кто реально нажал) — публично виден только ВП. Аудит доступен владельцу.
- Механика в API: запрос с параметром `acting_as={vu_id}`; сервер проверяет право (владелец/оператор) и пишет аудит.

## 3. Ключи (без собственных устройств)

```
Ключи ВП (identity + signing) создаёт устройство владельца при создании ВП
    → приватные ключи хранятся ТОЛЬКО wrapped на устройства владельца и операторов
Входящее сообщение к ВП: обычный конверт; wrapped_key на identity-ключ ВП
    → читает каждый, у кого есть wrapped-ключ ВП
Исходящее от ВП: подпись signing-ключом ВП; actor — в серверном аудите
Смена состава операторов → ротация ключей ВП
    (старый оператор гарантированно не читает новые сообщения)
```

Для отправителя чат с ВП — обычный «личный чат» ([03-personal-chat](../doc_UI/03-personal-chat.md)); в окне информации виден признак «отвечает команда». Подробно: [key-lifecycle.md](../03-security/key-lifecycle.md) §8.

## 4. Managed inbox (окно 4 «Социальное взаимодействие»)

Личные сообщения ВП — «сообщения группы» его команды: владелец и операторы получают их **карточками во «Входящих» окна 4**, находясь под основным логином. Обращение открывается как настоящий чат с панелью оператора (Взять / Отложить / Закрыть, виден ответственный). Статус и ответственный — серверные, общие для команды. Полная спецификация окна — [10-social-interaction](../doc_UI/10-social-interaction.md), модель карточек и правил — [data-model.md](../02-architecture/data-model.md) §11.

## 5. Передача (продажа)

Передача ВП другому пользователю — встроенная операция:

```
1. Владелец инициирует передачу → указывает получателя
2. Получатель подтверждает (двустороннее согласие)
3. Сервер: owner_user_id меняется, операторы отзываются, запись в аудит передач
4. Ключи: РОТАЦИЯ по умолчанию — новые ключи ВП создаются устройством нового
   владельца; будущие сообщения старому владельцу гарантированно недоступны.
   История: передаётся по договорённости (экспорт/ключи) — вне гарантии платформы
```

Вместе с ВП переходит **всё его имущество**: каналы, группы, сообщества, подписчики, рейтинг.

**Честные оговорки (фиксируются в пользовательском соглашении):**
1. Старый владелец мог сохранить копии истории и ключей до передачи — это технически неустранимо; гарантия распространяется только на сообщения **после** ротации.
2. Платформа фиксирует акт передачи (аудит); денежные расчёты — вне платформы. Встроенный маркетплейс — пост-MVP.

## 6. Лимиты и анти-абьюз (NFR)

| Параметр | MVP | Цель |
|----------|-----|------|
| ВП на полный аккаунт | 5 | настраиваемо |
| Операторов на ВП | 20 | 50 |
| Временный аккаунт | ВП недоступны | — |

- Создание ВП — только полный аттестованный аккаунт; rate-limit на создание.
- Имперсонация («Редакция канала X» посторонним) — общий механизм жалоб + приоритетная модерация для ВП.
- Escrow: сообщения ВП подпадают под общую политику ([escrow-legal-access.md](../03-security/escrow-legal-access.md)); аудит передач помогает при юридических запросах.
