# Ленты, атрибуты, жанры

> Каноническая модель формирования лент (окна 2–3): **общую ленту формирует сервер** (оценки + атрибуты), **ленту друзей формирует пользователь** (публичные полки избранного). Решения зафиксированы 2026-07-12. Схема — [data-model.md](../02-architecture/data-model.md) §7, UI — [35-attributes-genres](../doc_UI/35-attributes-genres.md).

## 1. Две ленты — два механизма

| | Общая лента | Лента друзей |
|---|-------------|--------------|
| Кто формирует | **Сервер** | **Пользователь** (курирование) |
| Источник | Публичный контент, прошедший одобрение в атрибутах | Публичные полки избранного друзей + их посты/репосты на стене |
| Ранжирование | Скоринг по сигналам (§4) + тематические срезы по атрибутам | Хронология |
| Управление | «Мои атрибуты», чипсы жанров/атрибутов, [+]/[−] | Состав друзей |

## 2. Полки избранного (страница пользователя, окно 5)

Избранное разделено на **две полки**; выбор полки — одним тапом при сохранении (⭐).

| Полка | Хранение | Кто видит | В ленту друзей |
|-------|----------|-----------|----------------|
| **Публичная** | Метаданные (plaintext) | Друзья на странице | ✅ питает её |
| **Личная** | **Шифруется**: список закладок = `SecretBox(shelf_key)`; ключ обёрнут на устройства владельца | Только владелец + те, кому выдан доступ | ❌ никогда |

**Доступ к личной полке — по запросу**: друг запрашивает → владелец одобряет → `shelf_key` заворачивается на identity-ключи устройств друга (wrapped keys, механика совместных коллекций — [crypto-protocol.md](../03-security/crypto-protocol.md)) → друг видит полку на странице владельца. Отзыв = ротация `shelf_key` + перевыдача оставшимся. Сервер видит только факт существования полки и список грантов, не содержимое.

## 3. Атрибуты и жанры

**Атрибуты = хэштеги** — единый реестр, второй сущности нет: `#программирование` в тексте поста (entity `hashtag`) резолвится в атрибут реестра.

```
ЖАНР (курируемая сервером «папка», их немного)
  └── АТРИБУТЫ (фолксономия: создают авторы, их очень много)
        └── посты (пары (пост, атрибут) со статусом одобрения)
```

**Жизненный цикл атрибута:**

1. Автор при публикации указывает атрибуты: автодополнение по реестру («похожий уже есть — использовать его?») или создание нового. Имя нормализуется (lowercase, trim) против дублей.
2. При создании нового атрибута автор **предлагает жанр**. Сервер: авто-принимает / рекомендует другой (по схожим атрибутам) / оставляет без жанра до ручной классификации. Кураторы могут переназначить жанр в любой момент — принадлежность жанру определяется **сервером**.
3. Дубли сливаются модерацией (merge: посты и подписчики переносятся).

**Обнаружение пользователем** (профиля интересов при онбординге нет):
- тап по атрибуту на посте → карточка атрибута → **[Добавить себе]**;
- глобальный поиск → тип «Атрибуты» ([17-global-search](../doc_UI/17-global-search.md));
- страница жанра → список атрибутов внутри.

«Мои атрибуты» → чипсы над общей лентой: жанры и добавленные атрибуты, тап = мгновенный тематический срез.

## 4. Одобрение контента в атрибуте («автор + репутация»)

Чтобы тематическая выдача не засорялась неверными метками, пара *(пост, атрибут)* имеет статус:

```
declared → approved | rejected
```

| Автор | Что происходит |
|-------|----------------|
| Канал/автор с хорошей репутацией (рейтинг «+/−», нет подтверждённых жалоб на метки) | `approved` автоматически |
| Новый / спорный автор | Выборочная проверка; до approve пост виден по подписке, но не в тематической выдаче |
| Подтверждённая жалоба «неверная метка» | `rejected` + штраф репутации автора |

Репутация считается от [rating_counters](../02-architecture/data-model.md) и истории жалоб (`reports`).

## 5. Сигналы ранжирования общей ленты

| Сигнал | Вес | Источник |
|--------|-----|----------|
| [+] / [−] пользователя | Сильный | `recommendations` |
| Эмоции (валентность ±1) | Средний | `emotions` |
| Мои атрибуты | Средний (тематический буст) | `user_attributes` |
| Подписки (каналы, сообщества) | Базовый (кандидаты) | `subscriptions` |
| Свежесть, глобальная популярность | Тай-брейк | `posts` counters |

Механика выдачи (MVP): кандидаты = подписки + approved-посты моих атрибутов + популярное в моих жанрах → скоринг-воркер → `feed:{user_id}` в Redis ([system-architecture.md](../02-architecture/system-architecture.md) §3.4). Поведенческие сигналы (досмотры, время чтения) — пост-MVP, только с согласия (см. аналитика, opt-in).

## 6. Лента друзей — механика

```
Друг добавил пост на ПУБЛИЧНУЮ полку (или репостнул на стену)
  → событие в fan-out → feed_friends:{user_id} каждого его друга
  → выдача хронологией с пометкой «⭐ от Алексея»
```

Никакого скоринга: пользователь сам курирует, чьи полки он видит (состав друзей). Личная полка в этот поток не попадает никогда (§2).
