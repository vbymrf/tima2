# Синхронизация и офлайн

> Модель: сервер — надёжный ретранслятор ciphertext + wrapped keys; **основная копия защищённой истории — устройства пользователя**. UI-контекст: [23-auth-recovery](../doc_UI/23-auth-recovery.md), [24-device-linking](../doc_UI/24-device-linking.md).

## 1. Источники истины

| Данные | Источник истины |
|--------|-----------------|
| Защищённые сообщения (plaintext) | Локальные БД устройств |
| Конверты + wrapped keys (≤ 90 дней) | Сервер |
| Публичный контент | Сервер |
| Статусы доставки/прочтения | Сервер (агрегирует от устройств) |
| Настройки, подписки, избранное | Сервер |
| Черновики, позиции прокрутки | Устройство |

## 2. Протокол синхронизации

Каждое устройство хранит `sync_cursor` (последний подтверждённый `event_id`). Подключение:

```
1. WS connect + auth (device JWT)
2. → sync.pull {cursor}
3. ← пакеты событий: message.new, receipt, key.rotated, chat.updated, ... (упорядочены по event_id)
4. Клиент применяет, подтверждает ack {event_id} → сервер сдвигает cursor
5. Далее — live-поток тех же событий
```

- События идемпотентны (повторное применение безопасно) — при обрыве docket просто перечитывается с cursor.
- Большой отрыв (устройство спало недели) → пагинация pull; сообщения старше 90 дней приходят без wrapped keys (читаются, только если уже были в локальной БД).

## 3. Офлайн-исходящие

```
Отправка без сети:
1. Сообщение шифруется сразу (конверт готов офлайн; wrapped keys — по кэшу ключей устройств получателя)
2. В локальную очередь outbox (SQLDelight) со статусом pending, UI показывает 🕓
3. При появлении сети — отправка по порядку; client_msg_id (UUID) обеспечивает дедупликацию на сервере
4. Если ключи устройств получателя устарели (новое устройство появилось офлайн) —
   сервер отвечает 409 stale_keys → клиент довёртывает недостающие wrapped keys и повторяет
```

## 4. Мультиустройство

- Каждое устройство — независимый подписчик событий со своим cursor.
- Прочтение на одном устройстве → событие `receipt.read` на остальные (снимает бейджи).
- Черновики и позиция прокрутки не синхронизируются (MVP).
- Новое устройство: bootstrap = профиль + список чатов + конверты и wrapped keys за 90 дней; более старая история — опциональный P2P-перенос с доверенного устройства зашифрованным каналом ([key-lifecycle.md](../03-security/key-lifecycle.md) §2).

## 5. Офлайн-чтение и кэш

- Локально всегда доступно: вся сохранённая история чатов, скачанные медиа, pinned посты/коллекции.
- «Скачать для офлайна» — на пост/коллекцию: медиа pin'уется в кэше и не вытесняется LRU ([media-storage.md](./media-storage.md) §7).
- Ленты: последняя загруженная страница читаема офлайн; действия (эмоции, [+]/[−], комментарии) складываются в outbox и уходят при появлении сети.

## 6. Конфликты

Модель почти бесконфликтна (append-only):

| Случай | Разрешение |
|--------|-----------|
| Одна эмоция с двух устройств | Last-write-wins по серверному времени |
| Редактирование профиля с двух устройств | Last-write-wins по полю |
| Ротация GK во время офлайна участника | Участник получает все пропущенные версии wrapped_GK при sync.pull |
| Отзыв устройства, у которого остался outbox | Сервер отклоняет (revoked); сообщения остаются локально с пометкой «не отправлено» |
