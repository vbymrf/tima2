# Развёртывание TIMA на хостинге — пошагово

Инструкция для запуска боевого сервера TIMA на одном VPS. Рассчитана на копирование команд по порядку. Весь стек (сервер, БД, хранилище медиа, звонки, escrow, TLS) поднимается одной командой `docker compose`.

Технический справочник с обоснованиями — [doc/07-deployment/server-setup.md](doc/07-deployment/server-setup.md). Здесь — практический путь «с нуля до работающего телефона».

---

## Что понадобится

| Что | Значение |
|-----|----------|
| **VPS** | 4 vCPU, 8 ГБ RAM, 80+ ГБ диск. Ubuntu 24.04. Провайдер любой (Hetzner, Timeweb, Selectel, aeza…) |
| **Публичный IPv4** | Без «серого» NAT — для звонков нужен реальный IP и открытый UDP |
| **Домен** | Любой, напр. `example.com`. Нужен для TLS (без него телефон будет ругаться на сертификат) |

> Без домена тоже можно (по IP, HTTP), но это только для быстрой проверки — телефон будет жаловаться на незащищённое соединение, а звонки могут не работать. Домен стоит ~200 ₽/год и снимает эти проблемы. Дальше инструкция — с доменом.

---

## Шаг 1. DNS: три записи на IP сервера

В панели, где куплен домен, создай три A-записи (замени `<IP>` на адрес VPS):

```
api.example.com   A   <IP>
s3.example.com    A   <IP>
lk.example.com    A   <IP>
```

Проверить, что разошлось (через 5–30 минут):
```bash
nslookup api.example.com
```
Должен показать твой `<IP>`.

---

## Шаг 2. Подготовка сервера

Подключись к VPS по SSH (`ssh root@<IP>`) и выполни:

```bash
# Пользователь для работы (не под root)
adduser deploy && usermod -aG sudo deploy

# Docker
curl -fsSL https://get.docker.com | sh
usermod -aG docker deploy

# Файрвол
ufw allow OpenSSH
ufw allow 80,443/tcp          # веб (TLS, ACME)
ufw allow 443/udp             # HTTP/3
ufw allow 7881/tcp            # LiveKit (fallback)
ufw allow 50000:60000/udp     # LiveKit медиа (звук/видео звонков)
ufw --force enable
```

Дальше работаем под `deploy`:
```bash
su - deploy
```

---

## Шаг 3. Забрать код на сервер

```bash
sudo mkdir -p /opt/tima && sudo chown deploy /opt/tima
git clone <URL-репозитория-TIMA> /opt/tima
cd /opt/tima/server
```

> `<URL-репозитория-TIMA>` — адрес твоего git (GitHub/GitLab/самохост). Если репозиторий приватный — сначала настрой SSH-ключ или токен доступа.

---

## Шаг 4. Секреты (`.env`)

Копируем шаблон и генерируем пароли:

```bash
cp deploy/.env.example deploy/.env
chmod 600 deploy/.env
```

Открой `deploy/.env` (`nano deploy/.env`) и заполни. Пароли сгенерируй командой (запускай по разу на каждое поле):
```bash
openssl rand -hex 24      # для POSTGRES_PASSWORD, REDIS_PASSWORD, MINIO_ROOT_PASSWORD
openssl rand -hex 32      # для JWT_SIGNING_KEY, LIVEKIT_API_SECRET
openssl rand -hex 16      # для LIVEKIT_API_KEY
```

Итоговый `.env` (пример):
```dotenv
DOMAIN=example.com
POSTGRES_USER=tima
POSTGRES_PASSWORD=8f3c...          # вставь сгенерированное
POSTGRES_DB=tima
REDIS_PASSWORD=a91b...
MINIO_ROOT_USER=tima-admin
MINIO_ROOT_PASSWORD=77de...
JWT_SIGNING_KEY=c40a...
LIVEKIT_API_KEY=1a2b...
LIVEKIT_API_SECRET=9f8e...
TIMA_DEV_SMS=1                      # для первого теста; перед пользователями → 0
```

---

## Шаг 5. Ключ LiveKit в конфиг звонков

Ключ и секрет LiveKit должны совпадать в двух местах. Впиши в `deploy/livekit/livekit.yaml` те же значения, что в `.env`:

```bash
nano deploy/livekit/livekit.yaml
```
Строку в секции `keys:` привести к виду `<LIVEKIT_API_KEY>: <LIVEKIT_API_SECRET>`, например:
```yaml
keys:
  1a2b3c...: 9f8e7d...
```

---

## Шаг 6. Первый запуск

Из каталога `/opt/tima/server`:

```bash
docker compose -f deploy/docker-compose.prod.yml --env-file deploy/.env up -d --build
```

Первая сборка идёт 2–5 минут (компилируется Go-бэкенд и escrow). Дальше проверь, что всё поднялось:
```bash
docker compose -f deploy/docker-compose.prod.yml ps      # все — healthy/running
```

**Сразу сохрани доли Шамира** — они печатаются один раз, при первой инициализации escrow-анклава:
```bash
docker compose -f deploy/docker-compose.prod.yml logs escrow-stub
```
Скопируй 5 строк `tima-escrow-share-v1:...` в надёжное место (менеджер паролей / офлайн). Они нужны для юридического доступа и **повторно не выдаются**.

---

## Шаг 7. Проверка

```bash
curl -fsS https://api.example.com/api/v1/healthz
```
Ответ `{"status":"ok"}` по HTTPS = сервер жив, сертификат Caddy получил автоматически.

> Если сертификат ещё не выдан — подожди минуту (Let's Encrypt) и проверь, что порт 80 открыт и DNS разошёлся.

---

## Шаг 8. Подключить телефон

1. Скопируй **`TIMA-0.1.0.apk`** (в корне репозитория) на телефон — через кабель, облако или мессенджер.
2. На телефоне разреши установку из этого источника и установи APK.
3. Открой TIMA. На экране входа в поле **«Сервер»** впиши:
   ```
   https://api.example.com
   ```
   (приложение само переключит WebSocket на `wss://`).
4. Введи номер телефона → нажми получить код. Поскольку `TIMA_DEV_SMS=1`, код придёт прямо в приложение (поле dev-кода) — введи его.
5. Готово: можно писать, звонить, создавать группы и каналы. Второй телефон/десктоп с тем же адресом сервера увидит переписку.

---

## Обновление сервера (новая версия кода)

```bash
cd /opt/tima && git pull
cd server
docker compose -f deploy/docker-compose.prod.yml --env-file deploy/.env up -d --build backend worker
```
Миграции БД применяются автоматически при старте бэкенда. Данные (БД, медиа, ключ escrow) лежат в docker volume и переживают пересборку.

## Отладка и память

Сводка runtime сервера (goroutines, heap) — не проброшена наружу, смотри изнутри контейнера:
```bash
docker compose -f deploy/docker-compose.prod.yml exec backend wget -qO- 127.0.0.1:6060/debug/stats
```
Если `goroutines` или `heap_alloc_mb` растут при простое между двумя запросами — это утечка, дальше снимаем профиль:
```bash
docker compose -f deploy/docker-compose.prod.yml exec backend wget -qO- 127.0.0.1:6060/debug/pprof/heap > heap.out
```

Память всего стека:
```bash
docker stats --no-stream
```

> На твоём ПК (dev) для того же — скрипт **`ОТЛАДКА-TIMA.bat`** в корне: показывает процессы, порты, дубли запусков и Docker/Vmmem; `-Clean` убирает мусор от повторных запусков.

## Перед реальными пользователями (чек-лист)

- [ ] `TIMA_DEV_SMS=0` в `.env` + подключён реальный SMS-провайдер (иначе код виден в ответе — небезопасно).
- [ ] Доли Шамира escrow сохранены офлайн у разных держателей.
- [ ] Настроены бэкапы PostgreSQL и MinIO (см. [server-setup.md §10](doc/07-deployment/server-setup.md)).
- [ ] `.env` не в git (уже в `.gitignore`), права `600`.
- [ ] Escrow-stub заменить на HSM — по плану фаза 6 ([escrow-legal-access.md](doc/03-security/escrow-legal-access.md) §7).

## Если что-то не так

| Симптом | Причина / решение |
|---------|-------------------|
| `healthz` не отвечает по HTTPS | DNS не разошёлся или порт 80 закрыт — Caddy не смог получить сертификат. Проверь `nslookup` и `ufw status` |
| Телефон: «не удалось подключиться» | Проверь адрес в поле «Сервер» (именно `https://api.…`), что healthz отвечает, что телефон в интернете |
| Код из SMS не приходит | Для теста должно быть `TIMA_DEV_SMS=1` — код придёт в приложение. Иначе нужен SMS-провайдер |
| Звонок соединяется, но нет звука | Закрыт UDP 50000–60000 или `use_external_ip` в livekit.yaml не `true` |
| `docker compose ps` — backend unhealthy | Смотри `docker compose ... logs backend`: чаще всего неверный `.env` (пароль БД) |
```
